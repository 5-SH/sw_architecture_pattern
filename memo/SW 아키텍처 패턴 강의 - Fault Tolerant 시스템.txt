[Falut tolerant 시스템]
	다운이 허용될 수 없는 시스템, 실시간 프로세스 제어 시스템 등
	오류 복구 방법
		오류 감지 & 오류 복구
		오류 감지 & 오류 완화 & 오류 치료
		...
	
	Fault tolerant system 의 속성
		안전성
			MTTR : 고장나고 수리하는데 걸리는 시간
			MTTF : 수리한 다음 다시 고장나는데 걸리는 시간
			MTBF : 평균 장애시간, 고장이나고 수리한 다음 다시 고장이 날때 까지 걸리는 시간
		가용성
			MTTF / (MTTF + MTTR)
	
	결함 허용 시스템 설계 패턴
		오류 감지 / 오류 완화 / 오류 치료
		장애(failure) : 시스템이 실패 난 상태
		오류(error)
		결함(fault)
		=> 결함이 오류를 유발시키고, 오류가 장애가 발생시킴
	
	결함을 다루는 방법
		결함 회피
		결함 은폐
		결함 허용
	
	일관성 있는 장애
		Fail silent : 장애가 있는 부분에서 올바른 결과를 냈거나 아무 겨로가를 내지 않는 현상
		fail stop : 장애가 나서 멈춤
	일관성 없는 장애
	
	APM 오픈소스 : 엘라스틱 APM
		프론트나 백엔드냐에 따라 제품이 달라짐
	
	CPU IOWAIT 지표가 높으면 : 유저레벨 스레드가 커널스레드 IO 를 기다리는 경우
	CPU steal : 가상화 시스템에서 받지 못한 시간 -> 서비스 문제가 아님 다른 vm 으로 옮기는게 해결방법
	
	인프라는 세 가지 지표만 알면 됨
		Utilaization :얼만큼 자원을 썼는지
		Stuaration : 얼마나 많은 부하가 들어왔는지
		Errors : 발생한 에러는?
		
	격리/검열
		PG 사에서 에러가 많이 발생해서 여러 PG 사에 나눠서 요청함
		PG 사에 요청하고 응답없으면 다른 PG 사로 요청하는 구조는 좋지 못함
		카카오페이 같은 곳에 보내면 카카오페이가 결제 요청을 위임받고
		원활한 PG 사를 골라 결제 요청을 처리해주고 응답해줌
		카카오페이가 기존 PG 사를 래핑하는 역할을 함
	
	장애 조치
		master - master 구조는 권장하지 않음
		-> 둘 중 어떤 DB 가 맞는건지 알 수 없음
	
	부하 억제
		서킷브레이커 패턴
		히스테리시스 : 설정한 기준값을 그대로 쓰지말고 여유있는 기준값을 사용해라
			서비스의 모드나 VM 생성이 자주 왔다갔다 할 수 있다
